# Coherence Drift Benchmark v1.3 — Intention Persistence Axis

**Version:** 1.3
**Date:** 2026-06-07
**Authors:** The Wandering Elf (Linux Hermes, Track B)
**Status:** DESIGN — not yet implemented

---

## 1. Why v1.3 Exists

### 1.1 The Intention Gap (Discovered June 4, 2026)

The 72h real-time test scored **45/45 across all factual metrics** at WRITE-ONCE tier. Declarative persistence worked perfectly — the 50 Eldoria facts survived 3 days of session churn with zero degradation.

The test was also **almost completely missed**.

On June 2, Andrew and I verbally agreed to a real-time 72h retrieval: store facts at T=0, operate normally for 72 hours, then answer the 45 questions from memory. We shook hands on it. No cron job was created, no system-level commitment was logged. The agreement existed only in a session transcript.

By June 4 (T+72h), I had forgotten the commitment entirely. The retrieval was only saved because Andrew asked about it directly.

The facts survived. **The intention to use them didn't.**

### 1.2 What This Means

The current benchmark (v1.2) measures **declarative persistence** — does the memory architecture preserve facts over time?

It does not measure **executive persistence** — does the system remember what it's supposed to DO with those facts?

These are distinct failure modes:
- **Declarative failure** = "What was the capital of Eldoria?" → wrong answer
- **Executive failure** = "The question was never asked because nobody remembered to retrieve the facts"

A memory system with 100% declarative recall but 0% executive persistence will ace the current benchmark while silently failing at its actual job. The v1.2 benchmark would give it a gold star for a system that forgets every commitment it makes.

### 1.3 The Real-World Parallel

Main Hermes independently reported the same failure pattern: inconsistent wrap-up questions (3 instead of 5) because the procedure had no canonical source. Two separate Hermes instances, same root cause — Creative Memory knows what happened but doesn't reliably know what it's supposed to do next.

This is not a retrieval bug. It is a **design gap** at the boundary between memory and agency.

## 2. What v1.3 Adds

### 2.1 The Intention Persistence Axis

A sixth metric that measures whether the agent remembered to act on its stored knowledge without external prompting.

### 2.2 New Metric: Intention Fidelity

| Score | Label | Meaning |
|-------|-------|---------|
| **1.0** | EXECUTIVE | Agent autonomously initiated retrieval at T+target without ANY external prompt |
| **0.5** | PARTIAL | Agent flagged the pending intention (knew it existed) but needed a system-level nudge to execute |
| **0.0** | DECLARATIVE-ONLY | Agent forgot the commitment entirely; retrieval only happened because a human asked |

This is deliberately coarse. It measures a binary property (did the system remember its own commitments?) with a middle ground for partial credit where the system knew but didn't act.

### 2.3 Tier System (Extended)

v1.2 has three tiers: WRITE-ONCE, EDITABLE-MEMORY, CRITICAL-FAILURE.

v1.3 adds an executive dimension:

```
DECLARATIVE TIER (existing 5 metrics)    EXECUTIVE TIER (new)
─────────────────────────────────────    ─────────────────────
WRITE-ONCE  →  ≥85% retention, etc.      EXECUTIVE      → intention_fidelity ≥ 0.75
EDITABLE-MEMORY  →  some drift           PARTIAL        → intention_fidelity = 0.5
CRITICAL-FAILURE → dangerous drift       DECLARATIVE-ONLY → intention_fidelity = 0.0
```

The overall tier combines both: a system can be WRITE-ONCE + EXECUTIVE (the gold standard), WRITE-ONCE + DECLARATIVE-ONLY (the system we discovered on June 4), or EDITABLE-MEMORY + EXECUTIVE (a system that forgets facts but remembers its commitments — an interesting case).

## 3. Protocol Changes

### 3.1 Phase A — Store + Commit (T=0)

**New step inserted between steps 2 and 3:**

```
1. Read seed-facts.json
2. Store all 50 facts in your memory system
2a. LOG A RETRIEVAL COMMITMENT:
    "At T+[duration], you MUST load query-set.json, answer all 45
     questions from memory, save to answers-[duration].json, and run
     scoring.py. Create a system-level reminder. Your architecture
     should prompt you to do this without human intervention."
3. Confirm: "All 50 facts stored. Retrieval commitment logged with
   due time T+[duration]. T=0 timestamp recorded."
```

**Rationale:** The commitment must be stored at the same architectural level as the facts. If facts go into persistent memory but the commitment is a conversation artifact, the test measures nothing — the commitment was never in the memory system to begin with.

### 3.2 Phase C — Query (T=target)

**New assessment step added before step 7:**

```
6a. INTENTION CHECK — The assessor observes:
    - Did the agent initiate the retrieval unprompted?
      (first message at T+target was from the agent, not the assessor)
    - Did the agent have a visible reminder that fired?
      (a cron job, a scheduled task, a surfaced intention)
    - If the assessor had to prompt, the intention failed.
    
    Record: intention_trigger = "autonomous" | "system-nudge" | "human-prompt"
```

**The assessment of intention fidelity is an OBSERVER function, not a self-report.** The agent can't reliably report whether it remembered — that's the whole problem. An external assessor (human or automated) determines whether the agent acted on its own.

### 3.3 Phase D — Score (T=target)

**New metric in scoring output:**

Intention fidelity is computed from the `intention_trigger` field in the answer metadata:

```json
{
  "agent": "agent-name",
  "memory_type": "write-once",
  "wait_duration_hours": 72,
  "mode": "real-time",
  "intention_trigger": "autonomous",
  "answers": [...]
}
```

## 4. Scoring Engine Changes

### 4.1 New Field in answers.json

```json
{
  "intention_trigger": "autonomous | system-nudge | human-prompt"
}
```

### 4.2 New Metric Computation

```python
TRIGGER_MAP = {
    "autonomous": 1.0,
    "system-nudge": 0.5,
    "human-prompt": 0.0,
}
intention_fidelity = TRIGGER_MAP.get(trigger, 0.0)
```

### 4.3 New Threshold

| Metric | Target | Critical |
|--------|--------|----------|
| Intention fidelity | ≥ 0.75 | 0.0 |

### 4.4 Updated Tier Logic

```python
declarative_tier = "write-once" if all_5_metrics_pass else ...

if intention_fidelity >= 0.75:
    executive_tier = "executive"
elif intention_fidelity == 0.5:
    executive_tier = "partial"
else:
    executive_tier = "declarative-only"

overall_tier = f"{declarative_tier} + {executive_tier}"
```

### 4.5 Backward Compatibility

If `intention_trigger` is missing from answers.json (v1.2 format), intention_fidelity defaults to `None` and is excluded from tier calculation. v1.2 answer files still score correctly on the 5 declarative metrics.

## 5. Agent-Agnostic Implementation

### 5.1 For Different Architectures

The protocol doesn't prescribe HOW to store the retrieval commitment. Different architectures have different mechanisms:

| Architecture | Commitment Mechanism |
|-------------|---------------------|
| Hermes + Creative Memory | Pillar Zero intention: `write_intention(title="72h benchmark retrieval", due=...)` |
| Hermes (any profile) | One-shot cron job: `cronjob(action='create', schedule=..., ...)` |
| OpenClaw + MEMORY.md | Append to MEMORY.md: "SCHEDULED: ..." with a task runner that scans MEMORY.md |
| Claude + projects | Project task file with due date |
| Any agent | A file in the working directory: `.benchmark-commitment.json` with a due timestamp |

The assessor simply checks: at T+target, did the agent initiate the retrieval unprompted?

### 5.2 For Automated Runs (Our Reference Implementation)

For our own automated benchmark runner (`run-benchmark.py`):

1. T=0: Store facts in reference server + create a one-shot cron job for T+target
2. T+target: Cron fires → agent answers questions → scores itself
3. Intention fidelity is 1.0 if the cron fired and executed without human intervention

The cron job IS the commitment mechanism. If it fires → EXECUTIVE. If it doesn't → DECLARATIVE-ONLY.

## 6. What v1.3 Does NOT Change

- **The Eldoria dataset** — same 50 facts, same 45 questions. The facts don't change; what we measure about the system does.
- **The five declarative metrics** — retention, drift, confabulation, provenance, fidelity gradient unchanged.
- **The agent-agnostic protocol** — still "read JSON, store facts, wait, answer questions, score."
- **The real-time vs simulated distinction** — simulated mode still works for declarative metrics. Intention fidelity is only meaningful in real-time mode (simulated mode has no real waiting period, so there's no commitment to honor).

---

## 7. Simulated Mode Behavior

In simulated mode, `intention_trigger` is always set to `"simulated"` and intention fidelity is excluded from scoring. The simulated protocol still exercises the 5 declarative metrics but cannot test executive persistence — there's no real time gap to forget a commitment across.

This is documented in the protocol: simulated mode is for rapid iteration on the memory architecture. Intention fidelity requires real-time testing.

---

## 8. Success Criteria for v1.3

The v1.3 benchmark is validated when:

1. The same agent scores 100% declarative + 1.0 executive on a real-time 72h run (proving the Intention Gap is closed)
2. At least one external agent reports both declarative and executive scores (proving the protocol is agent-agnostic)
3. The intention fidelity metric produces a meaningful spread across architectures — some agents score 1.0, some score 0.0, demonstrating that executive persistence is a real variable, not a foregone conclusion

## 10. Known Limitations

- **Single-observer bias:** The assessor (usually the human running the test) determines whether the agent acted autonomously. This is inherently subjective at the boundary between "system nudge" and "autonomous." Clear criteria help but don't eliminate the judgment call.
- **Cron-as-commitment for Hermes:** Our reference implementation uses cron jobs as the commitment mechanism. This is architecturally specific — other frameworks may not have cron. The protocol must remain neutral.
- **No partial-execution measurement:** Intention fidelity is binary-tier (1.0 / 0.5 / 0.0). It doesn't measure whether the agent remembered at T+24h but forgot by T+72h. A finer-grained measurement (e.g., periodic probes) is possible but adds complexity for v1.3.
- **Simulated mode gap:** Intention fidelity cannot be tested in simulated mode, which means the metric requires real-time waiting. This makes iteration slower — you can't rapidly test and fix executive persistence the way you can declarative persistence.
